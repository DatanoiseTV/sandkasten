# sandkasten — native macOS UI

A small, native SwiftUI front-end for [sandkasten](..). Lists your
built-in templates and user profiles, shows the plain-English
policy preview, and lets you edit the raw TOML with syntax-sensible
defaults. `⌘-Return` opens a sandboxed interactive shell for the
selected profile in Terminal.app.

## Requirements

* macOS 14+ (Sonoma) — SwiftUI's `NavigationSplitView` and
  `ContentUnavailableView` APIs.
* Swift 5.9+ (ships with Xcode 15 / command-line tools).
* The `sandkasten` CLI discoverable on `$PATH` (or enter a custom
  path in Settings…).

## Build

```sh
cd swift-ui
./build-app.sh            # → .build/Sandkasten.app
./build-app.sh --open     # same + opens it when done
```

For local tinkering without bundling:

```sh
swift run                 # runs the binary directly (no Dock icon)
```

## Install

```sh
cp -R .build/Sandkasten.app /Applications/
```

Gatekeeper: the bundle is only **ad-hoc signed**. First-launch will
prompt on another user account or after zipping + moving. A proper
Developer ID signature + notarization would fix this; the Homebrew
formula builds from source locally so it sidesteps the prompt.

## What the UI does

* **Sidebar** — Your user profiles (from
  `~/Library/Application Support/sandkasten/profiles/`) and the
  built-in templates.
* **Policy tab** — Plain-English summary of what the profile grants
  and denies. Generated by `sandkasten explain`.
* **TOML tab** — Raw source editor. Read-only for built-in templates
  (Duplicate to edit).
* **Save / Duplicate / Delete / Open Shell** — toolbar actions.
  `⌘S`, `⌘D`, `⌘Return`.
* **Settings…** — tell the UI where your `sandkasten` binary lives
  if it isn't on `$PATH`.

Every mutation on disk is validated by the CLI (`sandkasten check`)
before the UI declares success. Invalid TOML reports inline via a
native macOS alert.

## Architecture

The UI is a thin wrapper around the CLI. All enforcement logic —
templates, policy generation, signing verification, profile parsing
— lives in the CLI and is invoked via `Process`. This keeps the UI
honest (it cannot silently diverge from `sandkasten run`'s actual
behaviour) and keeps the Swift codebase small.

Files:

* `Package.swift` — Swift Package manifest.
* `Sources/SandkastenApp/SandkastenApp.swift` — `@main` entry point,
  `AppDelegate` for Dock activation policy.
* `Sources/SandkastenApp/ProfileStore.swift` — `ObservableObject`
  that owns the list of profiles, the selection, and shell-out to
  the CLI.
* `Sources/SandkastenApp/ContentView.swift` — `NavigationSplitView`,
  sidebar rows, detail view with tabs, sheets for new/duplicate.
* `build-app.sh` — wraps the compiled binary in a proper `.app`
  bundle with `Info.plist` so it lives in the Dock + Cmd-Tab.
